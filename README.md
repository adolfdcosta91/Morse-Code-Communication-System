    <article class="markdown-body entry-content p-3 p-md-6" itemprop="text"><h2><a id="user-content-morse-beeper" class="anchor" aria-hidden="true" href="#morse-beeper"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Morse Beeper</h2>
<h4><a id="user-content-overview" class="anchor" aria-hidden="true" href="#overview"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Overview</h4>
<p>A <strong>beeper</strong> is a wireless communication device that receives and sends messages to others using its internal transceiver. In this project, we design and prototype a device that resembles a beeper and is able to store contacts, send, and receive <strong>morse code</strong> messages.</p>
<p>This project involves interfacing with an <strong>LCD</strong> screen, a <strong>radio</strong> module, the <strong>EEPROM</strong>, and a <strong>buzzer</strong>. We also design and implement an easy to navigate user interface in a constrained 16x2 screen while managing external <strong>interrupts</strong> and handling data <strong>storage</strong>.</p>
<p>In the end, all devices that follow this guideline should be able to communicate as described in this document.</p>
<p><a target="_blank" rel="noopener noreferrer" href="https://raw.githubusercontent.com/xaviermerino/ECE2551-SoftHardDesign/master/MorseBeeper/beeper.png"><img src="https://raw.githubusercontent.com/xaviermerino/ECE2551-SoftHardDesign/master/MorseBeeper/beeper.png" alt="beeper" style="max-width:100%;"></a></p>
<p><strong>What you will need:</strong></p>
<ul>
<li>Arduino LCD Keypad Shield</li>
<li>Piezo Buzzer</li>
<li>NRF24L01+ module</li>
</ul>
<p><strong>In the process you will:</strong></p>
<ul>
<li>Interface with a <strong>Liquid Crystal Display</strong> (LCD) component.</li>
<li>Use a voltage divider as an <strong>analog input</strong> to detect button presses.</li>
<li>Interface with an <strong>NRF2401L+</strong> module via SPI to communicate with other modules.</li>
<li>Use <strong>interrupts</strong> to process incoming messages.</li>
<li>Interface with a piezo <strong>buzzer</strong> to notify user of events.</li>
<li>Interface with the Arduinoâs <strong>EEPROM</strong> to store long-term data that can be retrieved after a power cycle.</li>
<li>Use the <strong>Watchdog Timer</strong>'s jitter to generate random byte sequences.</li>
<li>Implement a beeper as a <strong>state machine</strong>.</li>
<li>Apply the concept of abstractions in C++ by defining <strong>classes</strong> and instantiating <strong>objects</strong> from those classes to promote re-usability and increased efficiency.</li>
<li>Use <strong>inheritance</strong> to extend Arduinoâs classes.</li>
<li>Use <strong>pointers</strong> to handle dynamic memory allocation.</li>
</ul>
<h4><a id="user-content-entropy-class" class="anchor" aria-hidden="true" href="#entropy-class"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Entropy Class</h4>
<p>If we compare our beeper to a cellphone you'll agree that we need to assign it a phone number. Imagine that your phone number is <code>321-123-4567</code>. If you were not guaranteed that this number belonged <strong>only</strong> to you, then someone else (who shares your number) could also be getting your calls and texts.</p>
<p><a target="_blank" rel="noopener noreferrer" href="https://raw.githubusercontent.com/xaviermerino/ECE2551-SoftHardDesign/master/MorseBeeper/alicebob.png"><img src="https://raw.githubusercontent.com/xaviermerino/ECE2551-SoftHardDesign/master/MorseBeeper/alicebob.png" alt="alicebob" style="max-width:100%;"></a></p>
<p>We use the term <strong>universally unique identifier (UUID)</strong> to refer to a number that is used to uniquely identify some entity. The probability that a <strong>UUID</strong> will be duplicated is not zero but it is low enough to be considered negligible.</p>
<p>We are going to use the <code>Entropy</code> class to generate a sequence of random bytes using the <code>Watchdog Timer</code>'s natural jitter. These bytes will become our unique identifier. You can find the class <a href="https://sites.google.com/site/astudyofentropy/file-cabinet/Entropy-v1.0.2.zip?attredirects=0&amp;d=1" rel="nofollow">here</a>.</p>
<h4><a id="user-content-nr24-class" class="anchor" aria-hidden="true" href="#nr24-class"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>NR24 Class</h4>
<p>The <code>nRF24L01+</code> is a single chip radio transceiver that operates in the 2.4 - 2.5 GHz band. It features ultra low power consumption and speeds up to 2 Mbps. We need to use a microcontroller to configure this radio through a <strong>Serial Peripheral Interface (SPI)</strong>. <p>The memory map consists of the following sections:</p>
<ol>
<li>
<p><strong>Flags:</strong> There are three flags in the memory map that get verified at every boot for <strong>integrity</strong> and <strong>schema</strong>.</p>
<ul>
<li>
<p><strong>Initialization Flag:</strong> Consists of three bytes (000 - 002) that spell <code>0xC0FFEE</code>. These bytes are set during the device's setup stage.</p>
</li>
<li>
<p><strong>Contact List Flag:</strong> Consists of two bytes (018 - 019) that spell <code>0xFACE</code>. These bytes are set during the device's setup stage and mark the beginning of the contact list related entries in the <strong>EEPROM</strong>.</p>
</li>
<li>
<p><strong>Message List Flag:</strong> Consists of two bytes (171 - 172) that spell <code>0xCA11</code>. These bytes are set during the device's setup stage and mark the beginning of the message list related entries in the <strong>EEPROM</strong>.</p>
</li>
</ul>
</li>
<li>
<p><strong>Counters:</strong> There are two counter entries in the <strong>EEPROM</strong>.</p>
<ul>
<li>
<p><strong>Contact Counter</strong>: Keeps track the number of contacts stored in the <strong>EEPROM</strong>. May hold values from 0 to 10.</p>
</li>
<li>
<p><strong>Message Counter</strong>: Keeps track the number of messages stored in the <strong>EEPROM</strong>. May hold values from 0 to 20.</p>
</li>
</ul>
</li>
<li>
<p><strong>Offsets:</strong> There is one offset entry in the <strong>EEPROM</strong>. According to the memory table above, the base address for the messages (or the address for the first message) is <code>174</code>. By adding an offset to this base address we can obtain the position of any message element relative to the first one. The offset entry can store up to <strong>8-bits</strong> of information allowing you to traverse from location <code>174</code> all the way to location <code>429</code>. This is useful to point to the next location available for saving a message.</p>
</li>
<li>
<p><strong>Contact Objects:</strong> The <strong>EEPROM</strong> stores up to 10 contact objects. These objects are <strong>15 bytes</strong> long and contain the contact's <strong>name</strong> and radio's <strong>UUID</strong>. The object located at address <code>003</code> contains information regarding the node and is set during setup time. Since we only have room for ten contacts, additional contacts must not be allowed. See the <code>Contact</code> class for more information.</p>
</li>
<li>
<p><strong>Message Objects:</strong> The <strong>EEPROM</strong> stores up to 20 message objects. These objects are <strong>13 bytes</strong> long and contain information on the sender, the receiver, the payload, and the payload's length. Since we only have room for twenty messages in the given design, we may need to reuse <strong>EEPROM</strong> locations if we receive more than twenty messages. See the <code>Message</code> class for more information.</p>
</li>
</ol>

<p>A <code>Message</code> object stores the <strong>UUID</strong> of the sender and the receiver, alongside with the payload and its length. This gives a <code>Message</code> object the size of <strong>13 bytes</strong>.</p>
<ol>
<li>
<p><strong>Sender's UUID:</strong> The sender's 40-bit <strong>UUID</strong>.</p>
</li>
<li>
<p><strong>Receivers's UUID:</strong> The receiver's 40-bit <strong>UUID</strong>. Used to verify that message was received by the intended party.</p>
</li>
<li>
<p><strong>Payload:</strong> The payload consists of <strong>morse code</strong> messages that may vary in size. Their size cannot exceed 16 characters (since that's the maximum amount we can display on the LCD screen). Morse code uses the <strong><code>.</code></strong> and the <strong><code>-</code></strong> symbols and we will represent these with <strong>zeroes</strong> and <strong>ones</strong> respectively. This allows us to represent each character with a bit instead of a byte. We need 2 bytes to represent up to 16 characters.</p>
</li>
<li>
<p><strong>Payload Length:</strong> Since a message can vary in <strong>length</strong>, the payload length tells us how many bits are valid to be interpreted as part of the message. We use 1 byte to store the payload's length.</p>
</li>
</ol>

<li><strong>LEFT</strong>: Erases the last character and moves cursor to the left.</li>

<li><strong>DOWN</strong>: Scrolls letter</li>

The arrows complete a blink cycle in a second.</p>
<ul>
